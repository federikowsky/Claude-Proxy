<objective>
Add configurable CORS support and propagate provider rate-limit headers
(Retry-After, X-RateLimit-*) to clients.

Purpose: CORS enables browser-based clients (Open WebUI, custom dashboards) to call
the proxy directly. Rate-limit propagation lets clients implement intelligent backoff
instead of hammering a rate-limited proxy.

Output: CORS config + middleware wiring, Retry-After header propagation in error handler, tests.
</objective>

    **In `llm_proxy/domain/errors.py`:**

    Extend `ProviderHttpError.__init__` to accept an optional `retry_after` parameter:
    ```python
    class ProviderHttpError(BridgeError):
        def __init__(
            self,
            message: str,
            *,
            upstream_status: int,
            provider: str,
            retry_after: float | None = None,
        ) -> None:
            translated_status = ...  # existing logic
            details = {
                "provider": provider,
                "upstream_status": upstream_status,
            }
            if retry_after is not None:
                details["retry_after"] = retry_after
            super().__init__(message, status_code=translated_status, details=details)
    ```

    **In `llm_proxy/api/errors.py`:**

    Modify the BridgeError handler to propagate Retry-After when present in details:
    ```python
    @app.exception_handler(BridgeError)
    async def handle_bridge_error(request, exc):
        headers = {}
        retry_after = exc.details.get("retry_after")
        if retry_after is not None:
            headers["Retry-After"] = str(int(retry_after))
        return JSONResponse(
            content=exc.to_payload(),
            status_code=exc.status_code,
            headers=headers or None,
        )
    ```

    **In provider HTTP error paths** (openai_compat.py, anthropic.py, etc.):
    When receiving a response with status 429, parse the `Retry-After` header
    and pass it to `ProviderHttpError`:
    ```python
    retry_after_raw = response.headers.get("retry-after")
    retry_after = float(retry_after_raw) if retry_after_raw else None
    raise ProviderHttpError(
        message,
        upstream_status=status,
        provider=self._provider_name,
        retry_after=retry_after,
    )
    ```

    **Important:** Only parse numeric Retry-After (seconds). Do NOT parse HTTP-date format
    (too complex, rarely used by LLM providers). If parsing fails, set retry_after=None.
    Wrap float() in try/except ValueError.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → CORS | Cross-origin access control |
| provider → Retry-After → client | Rate-limit header forwarding |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-11 | S (Spoofing) | CORS | mitigate | Default to disabled. When enabled, require explicit allowed_origins list. Default wildcard is convenience for local dev only — docs will warn against production wildcard. |
| T-09-12 | T (Tampering) | Retry-After parsing | mitigate | Only parse numeric Retry-After. Wrap in try/except. Invalid values result in None (no header propagated). No HTTP-date parsing to avoid complexity. |
| T-09-13 | I (Info Disclosure) | Retry-After | accept | Retry-After is a standard HTTP header. Propagating it reveals no internal information beyond what the provider already communicated. |
</threat_model>
